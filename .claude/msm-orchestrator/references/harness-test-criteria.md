# Harness Test Criteria — MSM Portal 하네스 테스트 기준

하네스 품질을 측정하고 개선하는 3가지 기준과 테스트 방법론.

## 기준 1: 실행 가능성 (Executability)

**질문:** "에이전트가 이 지시를 따라갈 때 실제로 동작하는가?"

**왜 중요한가:** 에이전트는 사람과 달리 "아마 이렇게 하면 되겠지"라는 추론을 잘 못한다. 명시된 명령어가 정확히 동작해야 한다.

**우선순위:** CRITICAL — 실행이 안 되면 아무리 좋은 설계도 무의미

### 체크리스트
- [ ] 모든 CLI 명령어가 정확한 경로에서 실행 가능한가?
- [ ] 참조된 파일/스크립트가 실제로 존재하는가?
- [ ] package.json 스크립트가 정의되어 있는가?
- [ ] 환경 전제조건(mock 서버, 빌드 도구)이 명시되어 있는가?
- [ ] 실패 시 폴백 경로가 정의되어 있는가?

### 테스트 방법
1. 각 SKILL.md에서 `bash` 코드 블록을 모두 추출
2. 프로젝트 루트에서 각 명령어 실행
3. 종료 코드와 출력 확인
4. 참조된 모든 파일 경로를 `ls`로 존재 확인

### 합격 기준
- 모든 CLI 명령어 실행 성공 (exit code 0 또는 예상된 에러)
- 모든 참조 파일 존재
- 환경 전제조건 미충족 시 명시적 건너뛰기 로직 존재

## 기준 2: 라우팅 정확성 (Routing Precision)

**질문:** "사용자 요청이 올바른 스킬로 도달하는가?"

**왜 중요한가:** 스킬 트리거는 description 필드만으로 결정된다. 본문이 아무리 좋아도 description에 키워드가 없으면 트리거되지 않는다. 2개 스킬에 같은 키워드가 있으면 비결정적 라우팅이 발생한다.

**우선순위:** HIGH — 올바른 에이전트에 도달하지 못하면 품질 저하

### 체크리스트
- [ ] 각 스킬의 should-trigger 쿼리가 8개 이상 커버되는가?
- [ ] 한국어 격식/비격식/구어체 모두 커버되는가?
- [ ] 영어 변형이 포함되어 있는가?
- [ ] 후속 요청("다시 해줘", "수정해줘")이 커버되는가?
- [ ] 크로스 스킬 충돌이 없는가? (동일 키워드 2개 스킬)
- [ ] 각 스킬에 경계 진술(boundary statement)이 있는가?
- [ ] orchestrator가 서브스킬 키워드를 복사하지 않는가?

### 테스트 방법

#### Should-Trigger (스킬당 8-10개)
```
격식 한국어:  "페이지를 생성해주세요"
비격식 한국어: "페이지 만들어줘"
구어체:       "화면 좀 하나 만들어줘"
영어:         "create a new page"
암시적:       "publisher 목록이 필요해"
후속:         "다시 해줘", "수정해줘"
```

#### Should-NOT-Trigger (스킬당 8-10개)
- 키워드가 유사하지만 다른 스킬이 적합한 "near-miss" 쿼리
- 경계가 모호한 쿼리 (예: "컴포넌트 추가" → 코드? JSON?)

#### Cross-Skill Collision
- 모호한 쿼리를 모든 스킬 description에 대조하여 어디서 트리거되는지 확인
- 2개 이상에서 트리거되면 충돌 → 경계 진술 필요

### 합격 기준
- 각 스킬 트리거 점수 7/10 이상
- 크로스 스킬 충돌 0건
- 모든 스킬에 경계 진술 존재

### 개선 전략
- **경계 진술 추가**: "이 스킬은 X만 다룬다, Y는 다른 스킬을 사용할 것"
- **orchestrator 전용 트리거**: 복합 작업("만들고 검증까지") 키워드만 보유
- **동의어 커버리지**: "올려줘"="높여줘", "린트"="lint", "봐줘"="확인해줘"

## 기준 3: 계약 명확성 (Contract Clarity)

**질문:** "에이전트 간 핸드오프에서 무엇을 주고받는지 명확한가?"

**왜 중요한가:** 에이전트 팀에서 가장 취약한 지점은 에이전트 간 경계면. 한 에이전트의 출력이 다음 에이전트의 입력과 정확히 맞아야 파이프라인이 작동한다.

**우선순위:** MEDIUM — 실행과 라우팅이 맞아도 핸드오프가 불명확하면 품질 저하

### 체크리스트
- [ ] 각 에이전트의 입력/출력이 명시되어 있는가?
- [ ] 에이전트 A의 출력 형식이 에이전트 B의 입력 형식과 일치하는가?
- [ ] 반려/재요청 시 전달 형식이 양측에 동일하게 정의되어 있는가?
- [ ] 파일 기반 전달 시 경로 컨벤션이 정의되어 있는가?
- [ ] 실패 시 에스컬레이션 경로가 명확한가?

### 테스트 방법

#### 핸드오프 추적
각 파이프라인에서 에이전트 간 데이터 전달을 추적:
```
builder 출력 → reviewer 입력: 일치하는가?
reviewer 반려 → builder 입력: 형식이 정의되어 있는가?
reviewer 통과 → test-writer 입력: 최종 파일 상태가 명확한가?
```

#### 실패 경로 추적
각 에이전트 실패 시:
- 몇 회 재시도하는가?
- 재시도 후에도 실패하면 누가 결정하는가?
- 사용자에게 무엇을 보고하는가?

### 합격 기준
- 모든 파이프라인에서 에이전트 A의 출력 → 에이전트 B의 입력이 일치
- 반려/재요청 형식이 양측 에이전트에 동일하게 정의
- 모든 실패 경로에 에스컬레이션 정의

## 테스트 루프 프로세스

```
Step 1: 실행 가능성 테스트
  → CLI 명령어 실행, 파일 존재 확인
  → CRITICAL 이슈 발견 시 즉시 수정

Step 2: 라우팅 정확성 테스트
  → 트리거 시뮬레이션 (should-trigger / should-NOT-trigger)
  → 충돌 발견 시 경계 진술 추가

Step 3: 계약 명확성 테스트
  → 드라이런 시나리오 추적
  → 핸드오프 불일치 발견 시 양측 에이전트 수정

Step 4: 수정 적용
  → 발견된 문제를 일반화하여 수정 (특정 예시에만 맞는 좁은 수정 금지)

Step 5: 재검증
  → 수정된 파일 전체를 처음부터 다시 검증
  → 모든 기준 합격 시 완료, 아니면 Step 1로
```
